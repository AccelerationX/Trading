# TradingSystem 优化设计蓝图

## 1. 文档目标

这份文档回答一个核心问题：

`TradingSystem` 要怎样继续设计，才能逐步逼近“综合技术、基本面、情绪、政策、宏观事件，并最终形成高质量交易决策”的目标。

本文不是功能清单，而是下一阶段的总设计蓝图。它关注：

- 这套系统最终应该成为什么
- 当前版本离目标还差什么
- 哪些能力必须补齐
- 各种分析应该如何融合成单一决策逻辑
- 小资金高收益目标会怎样改变系统设计
- 各能力之间应如何组织，而不是互相堆叠
- 后续开发应该按什么顺序推进

## 2. 目标重新定义

### 2.1 真正目标

系统的真正目标不是“预测所有涨跌”，也不是“生成很多股票建议”，而是：

- 持续识别当日或未来数日最有交易价值的市场环境
- 识别最有持续性的主线、事件和资金方向
- 在账户约束下筛出最值得交易的少数机会
- 提供清晰、可执行、可复盘的计划
- 通过长期评估，把无效模块逐步淘汰，把有效模块保留下来

这里必须明确一个现实前提：

- 当前系统服务的是小资金账户
- 目标不是低波动稳健跑赢指数
- 目标更接近“在可承受较高波动的前提下，寻找高赔率机会”

这会直接影响：

- 候选风格选择
- 主题偏好
- 仓位分配方式
- 风险容忍度设定
- 对大票慢行情和小票爆发机会的取舍

### 2.2 不应误判的目标

系统不应该被设计成：

- 自动交易黑盒
- 靠一次宏观新闻就直接给出重仓结论的“新闻反应器”
- 单纯靠技术形态选股的监控器
- 只会写摘要、不会给出交易约束的研究助手

### 2.3 成功标准

如果未来系统足够成熟，成功标准应是：

- 每日输出数量不多，但重点明确
- 对为什么看多、为什么回避、为什么只观察，有稳定证据链
- 能区分“重要事件”和“无效噪音”
- 能区分“事件存在”和“交易机会成立”这两个不同层次
- 在 30 到 90 个交易日评估窗口里，表现优于无筛选、纯技术、纯主观三类基准
- 在小资金账户约束下，能持续找到高赔率机会，而不是只给出看起来正确但收益弹性不足的大票建议

## 3. 当前系统现状判断

### 3.1 已经具备的能力

当前系统已经有了一个可用底座：

- 市场环境初步判断
- 公司公告事件结构化
- 政策/主题卡片
- 资金行为卡片
- 候选股评分
- 交易计划卡片
- 账户约束映射
- 盘前摘要输出
- 复盘记忆沉淀

### 3.2 当前最关键的问题

当前版本距离最终目标，主要差在五个层面：

#### A. 缺少宏观事件层

系统能处理公司公告和政策主题，但还不能稳定处理：

- 国际访问
- 关税变化
- 外交缓和/对抗
- 地缘风险
- 大宗商品冲击
- 海外科技制裁
- 海外央行和美元流动性变化

这意味着它对“特朗普访华”这类重大事件目前没有专门结构，也没有稳定映射链路。

#### B. 缺少真正的情绪层

现在已有一些市场环境字段，但仍偏静态与结果型。

真正的 A 股情绪分析不只是：

- 涨跌家数
- 涨停数量
- 成交额

还需要：

- 龙头次日溢价
- 炸板质量
- 高位股承接
- 主线扩散与退潮
- 一致性过热与分歧回流
- 日内情绪修复质量

#### C. 事件与交易之间缺一层“解释引擎”

现在系统能识别“有事件”，但还不能稳定回答：

- 这个事件究竟影响什么方向
- 是情绪刺激、基本面改善，还是估值修复
- 利好是直接链条还是间接链条
- 持续性是一日脉冲还是多日主线
- 市场是否已经提前交易过

#### D. 基本面层太薄

当前更偏事件驱动和技术确认，对基本面只做了很弱的隐含使用。

后续至少需要有一层轻量但有用的基本面框架，用来区分：

- 题材空转票
- 有业绩/订单/产业景气支持的票
- 事件逻辑成立但股价不值得追的票

#### E. 运行模式不够分层

当前主流程容易把大量 LLM 任务绑进同一条运行链路，导致：

- 系统稳定性下降
- 日常运行耗时过长
- 主流程和增强流程耦合过重

这会妨碍系统成为真正可靠的日常工具。

#### F. 多分析融合逻辑仍不够清晰

当前系统虽然已经同时使用了：

- 技术分析
- 事件分析
- 主题分析
- 资金分析
- 账户约束

但它们之间仍更像“并列加权”，还不是一个足够清晰的因果决策链。

真正高质量的融合逻辑不应是：

- 每个模块给一点分，最后做平均

而应是：

- 先判断市场允许什么机会出现
- 再判断当前最值得交易的驱动是什么
- 再判断哪类股票最可能承接这个驱动
- 再判断具体股票是否有资金、基本面和技术确认
- 最后判断对这个账户来说值不值得做

这意味着后续必须设计一个明确的“分析融合引擎”。

## 4. 下一代系统设计原则

### 4.1 先做“决策质量提升”，不先做“功能堆积”

不是接更多数据源就更强，必须优先增加能显著改变决策质量的层：

- 事件解释层
- 情绪层
- 主题映射层
- 评估层

### 4.2 信息优先，技术确认

下一代系统的逻辑顺序应该是：

1. 先看发生了什么
2. 再看这些事对市场意味着什么
3. 再看哪些主线真正成立
4. 再看哪些股票是直接受益者
5. 最后才用技术模块做确认和筛选

即：

- `信息决定方向`
- `情绪决定可交易性`
- `技术决定执行点`

在这个框架里，基本面与资金面的位置也需要明确：

- `信息决定方向`
- `情绪决定环境`
- `基本面决定逻辑质量`
- `资金决定市场是否正在兑现这条逻辑`
- `技术决定入场与退出时机`
- `账户约束决定最终是否执行`

### 4.3 事件不直接等于交易

任何重大新闻必须经过四层判断：

1. 事件是否真实且重要
2. 逻辑链是否清晰
3. 市场是否认可
4. 个股是否适合这个账户交易

只有四层都通过，才能从“事件”升级到“计划”。

### 4.4 默认不承诺收益，承诺可评估

系统设计上不能承诺“高收益”，因为那不是设计目标可以直接保证的。

但系统必须承诺：

- 每个决策可追溯
- 每个模块可单独评估
- 每类建议可长期比较优劣

只有这样，系统才可能逐步靠近高收益。

### 4.5 可落地优先于完美设想

设计时必须坚持：

- 先做能每天稳定跑起来的系统
- 再做更强的解释和增强
- 不接受“理论上能做、实际上跑不动”的假系统

所以每个阶段都必须同时满足：

- 有明确输入
- 有明确输出
- 有明确运行时长预期
- 有明确失败降级方案
- 有明确可测试路径

### 4.6 不回避 LLM 问题，而是工程化解决

LLM 调用问题不能通过“少用就算了”来逃避，而应正面设计。

后续应明确解决：

- 哪些任务必须用 LLM
- 哪些任务不该用 LLM
- 如何限制 workpack 数量
- 如何做批处理、缓存、去重和重试
- 本地模型、远程模型、人工增强三者如何分工
- 当 LLM 不可用时主流程如何降级

目标不是减少 LLM，而是让 LLM 成为稳定、可控、可证明有价值的子系统。

## 5. 目标架构

建议将下一代系统明确拆成九层。

## 5.1 数据接入层

职责：

- 采集市场、公告、政策、新闻、宏观、资金、账户、人工输入
- 记录发布时间、获取时间、市场阶段、来源等级

建议新增数据族：

- `macro_news_wire`
- `global_market_watch`
- `commodity_and_fx_watch`
- `manual_macro_event_notes`

来源分级：

- `S`：官方、交易所、上市公司、官方统计
- `A`：Reuters、AP、Bloomberg、权威财经媒体
- `B`：二手整理平台
- `C`：社交传闻

规则：

- `S/A` 才能进入主分析流
- `B` 只能辅助
- `C` 默认不触发交易建议

## 5.2 市场状态层

职责：

- 分析大盘指数趋势
- 判断市场是否适合交易
- 判断当前偏进攻、轮动、兑现还是防守

下一步应强化为两个子模块：

### A. 指数趋势状态

针对：

- 上证指数
- 深成指
- 创业板指
- 沪深 300
- 中证 1000 / 2000

输出字段建议：

- `trend_state`: up / down / range / failed_breakout
- `trend_horizon`: intraday / swing / medium_term
- `trend_quality`: clean / noisy / weak
- `style_dominance`: large_cap / small_cap / balanced

### B. 投机情绪状态

输出字段建议：

- `sentiment_cycle`: expansion / rotation / contraction / panic_repair
- `leader_stability`
- `speculation_temperature`
- `follow_through_quality`
- `failure_pressure`

这层将决定：

- 哪类策略能出手
- 候选股阈值应该高还是低
- 是否只允许观察，不允许主动开仓

## 5.3 宏观事件层

这是当前最欠缺、也最重要的新层。

职责：

- 处理国际、宏观、跨市场重大事件
- 把“新闻标题”转成可交易的宏观框架

建议新增对象：

- `MacroEventCard`

字段建议：

- `event_id`
- `event_title`
- `event_category`
- `publish_time`
- `source_tier`
- `countries_or_regions`
- `macro_direction`
- `market_impact_scope`
- `a_share_impact_guess`
- `beneficiary_sectors`
- `risk_sectors`
- `duration_guess`
- `confirmation_needed`
- `confidence`

典型事件类型：

- 中美关系变化
- 关税/出口限制
- 能源与商品冲击
- 海外央行和流动性
- 地缘冲突
- 海外科技政策

## 5.4 主题与行业映射层

职责：

- 把政策、宏观事件、公司公告映射成交易主线
- 区分直接受益、间接受益、伪受益

建议强化为：

- `ThemeBoard`
- `SectorImpactMap`

核心输出应不是平铺股票，而是：

- 主线名称
- 驱动来源
- 逻辑链
- 龙头候选
- 中军候选
- 跟风观察
- 明确回避的弱跟风

## 5.5 个股案例层

建议引入比当前 `CandidateCard` 更完整的一层：

- `StockCase`

它不只是一个分数，而是一个个股交易案例。

字段建议：

- `stock_code`
- `case_type`
- `event_support`
- `theme_support`
- `macro_support`
- `capital_support`
- `fundamental_support`
- `technical_confirmation`
- `tradeability`
- `contra_risks`
- `why_now`
- `why_not_now`

这个对象的目标是回答：

- 为什么是这只
- 为什么是现在
- 为什么不是别的票

## 5.6 分析融合引擎

这是后续最关键的新设计之一。

职责：

- 把技术、基本面、情绪、信息、资金、账户约束变成统一决策链
- 防止系统退化成“多指标打分器”

建议输出对象：

- `IntegratedTradeHypothesis`

其核心不是分数，而是结构化判断：

- `market_permission`
  - 当前市场允许什么类型机会
- `primary_driver`
  - 本次机会的第一驱动是什么
- `driver_quality`
  - 这条驱动是否足够强
- `confirmation_state`
  - 资金、技术、主题是否确认
- `account_fit`
  - 对当前账户是否适配
- `expected_edge_type`
  - 是情绪脉冲、事件延续、趋势加强，还是修复博弈
- `expected_holding_window`
  - 预期持有窗口
- `actionability`
  - 是否能升级为交易计划

建议固定采用如下逻辑顺序：

### Step 1: 市场许可

先判断今天能不能做、适合做什么：

- 高弹性题材追击
- 事件驱动试错
- 趋势延续
- 低吸修复
- 纯观察

### Step 2: 驱动识别

再判断这只票上涨的主驱动是什么：

- 宏观驱动
- 政策驱动
- 公司事件驱动
- 基本面改善驱动
- 资金博弈驱动
- 纯技术驱动

规则：

- 纯技术驱动在小资金高收益系统里可以保留
- 但优先级通常低于“事件/主题/资金共振”

### Step 3: 逻辑质量判断

这一步决定“是不是值得认真看”。

看：

- 逻辑是否清晰
- 是否存在直接受益链
- 是否已有反向风险
- 是否已经过度交易
- 是否具备持续性

### Step 4: 确认状态判断

看：

- 题材是否发酵
- 龙头是否出现
- 成交是否承接
- 技术是否支持执行

### Step 5: 账户适配判断

小资金体系里，这一步不是保守过滤，而是收益效率过滤。

不仅要问：

- 能不能买

还要问：

- 值不值得用你的小资金去买

即：

- 低弹性大票虽然安全，但若收益空间不足，也不应占用有限出手机会
- 高弹性机会即使波动更大，只要逻辑清晰且仓位可控，优先级可以更高

### Step 6: 动作生成

最后才输出：

- avoid
- observe
- prepare
- buy_pilot
- add_on_confirmation
- reduce_or_exit

这个引擎将决定系统内在逻辑是否真正统一。

## 5.7 基本面支持层

不建议一开始做成重财务模型，但必须建立“轻基本面支持层”。

目标不是估值定价，而是过滤掉明显质量差的事件票。

建议先接入：

- 业绩预告/快报
- 营收利润增速方向
- 订单/合同性质
- 市值与流通盘
- 行业景气标签
- 是否存在连续减持、监管、亏损等压制

输出建议：

- `fundamental_quality`
- `earnings_support`
- `valuation_stretch_warning`
- `balance_sheet_risk_flag`

## 5.8 执行决策层

这一层保留现有框架，但要改成更明确的三步：

1. `can_trade`
2. `worth_trade`
3. `how_to_trade`

其中：

- `can_trade` 由账户、流动性、价格、仓位决定
- `worth_trade` 由事件、情绪、主题、资金、基本面决定
- `how_to_trade` 才由技术确认和具体执行规则决定

最终计划动作建议统一为：

- `avoid`
- `observe`
- `prepare`
- `buy_pilot`
- `add_on_confirmation`
- `reduce_or_exit`

### 小资金高收益账户的特殊原则

这套系统服务的小资金账户，不应沿用传统大资金稳健框架。

需要明确写进系统逻辑的原则：

- 更重视机会密度，而不是覆盖面
- 更重视弹性和赔率，而不是低波动
- 更能容忍个股波动，但不能容忍逻辑不清和流动性失真
- 更应优先围绕主线龙头、前排、中军，而不是低波动边角票
- 更应避免把有限交易机会浪费在“看起来合理但不够赚钱”的标的上

因此账户配置应新增：

- `return_target_style`
- `risk_tolerance_mode`
- `preferred_opportunity_type`
- `max_consecutive_loss_tolerance`
- `high_beta_allowed`
- `leader_preference`

系统后续应允许你明确指定：

- 你要的是稳健、均衡，还是高弹性高收益

然后让整套候选排序随之变化。

## 5.9 评估与记忆层

这是系统能否越做越强的关键。

未来必须不再只评估“最终结果”，而是分层评估：

- 市场状态判断是否有效
- 宏观事件判断是否有效
- 主线识别是否有效
- 候选生成是否有效
- 回避名单是否有效
- LLM 增强是否真的提升效果

建议新增：

- `RecommendationOutcomeRecord`
- `ModuleAttributionReport`
- `ThemeLifecycleReview`

## 6. 三种运行模式

为了让系统稳定且可扩展，建议未来固定区分三种运行模式。

### 6.1 日常稳定模式

特点：

- 不依赖重型 LLM
- 盘前 1 到 3 分钟内应可完成
- 主要靠结构化规则和轻量增强

用途：

- 每日固定盘前跑

### 6.2 事件增强模式

特点：

- 遇到重大新闻或重大政策时触发
- 专门为少数关键事件生成深度分析
- 可以调用更强的 LLM、更多外部信息源和更复杂的映射链路

用途：

- 如“特朗普访华”“关税调整”“半导体制裁升级”

### 6.3 复盘研究模式

特点：

- 不追求速度
- 用于盘后总结、模块比较、经验沉淀

用途：

- 优化系统本身

这三种模式必须解耦，不能继续把所有 LLM 工作默认塞进主流程。

## 6.4 LLM 子系统设计原则

后续 LLM 部分不再作为“附带增强”，而是作为独立子系统设计。

建议分成四类任务：

### A. 必须用 LLM 的任务

- 宏观事件解释
- 政策到行业链条映射
- 文本型公告深度解释
- 主题受益链推演

### B. 不该默认用 LLM 的任务

- 市场宽度统计
- 技术指标计算
- 账户约束过滤
- 一般候选排序的基础计算

### C. 可选增强任务

- 候选诊断润色
- 交易计划补充观察点
- 复盘记忆总结

### D. 人工覆盖任务

- 重大新闻手工标注
- 特殊题材人工指定观察方向

工程要求：

- workpack 必须分级
- 主流程必须限额
- 结果必须缓存
- 重复文本必须去重
- 批量任务必须允许中断和续跑
- LLM 不可用时必须自动降级

设计目标是：

- LLM 出问题时，系统仍然可用
- LLM 正常时，系统明显更强

## 7. 以“特朗普访华”为例的目标处理方式

未来系统遇到这类事件，理想处理流程应是：

### Step 1: 事件采集

从以下来源进入：

- 官方通稿
- Reuters / AP 等权威媒体
- 人工补录重点观察

### Step 2: 生成 MacroEventCard

系统不直接问“利好哪些股票”，而先问：

- 是外交礼仪事件，还是实质性经贸突破
- 是风险偏好改善，还是预期落空
- 哪些行业有一阶影响
- 哪些行业只有二阶想象

### Step 3: 生成影响映射

输出：

- 直接受益方向
- 潜在受压方向
- 需要市场确认的方向

例如可能涉及：

- 出口链
- 航空旅游
- 港口航运
- 农业进口链
- 科技制裁缓和预期链

### Step 4: 加入市场确认条件

系统必须附加：

- 指数是否认可
- 主线是否发酵
- 龙头是否出现
- 是否有资金承接

### Step 5: 才进入个股层

若没有市场确认，则结论只能是：

- 值得观察
- 不足以下交易结论

而不是直接生成买入建议。

这就是未来系统应有的判断方式。

## 8. 开发优先级

建议按四个阶段推进。

## Phase 1: 稳定化与架构重整

目标：

- 让系统先成为可靠工具

任务：

- 将 LLM 增强从主流程中解耦
- 主流程默认只跑稳定层
- 增加事件增强模式
- 修正运行超时和大量 workpack 问题
- 明确主流程与 LLM 子系统边界
- 为后续分析融合引擎预留对象结构

完成标准：

- 日常模式可稳定在数分钟内完成
- 即使无 LLM 也能产出完整盘前摘要

## Phase 2: 情绪层和指数趋势层

目标：

- 让系统真正知道“今天是什么市场”

任务：

- 增加指数趋势判断
- 增加投机情绪质量指标
- 将情绪状态直接影响候选阈值和动作建议
- 将小资金高收益偏好纳入候选排序逻辑

完成标准：

- 系统能明确区分“可积极试错”和“只能观察防守”

## Phase 3: 宏观事件层与主题映射层

目标：

- 让系统具备处理重大外部事件的能力

任务：

- 引入 MacroEventCard
- 补宏观与国际新闻源
- 增加事件到行业、行业到个股的映射规则
- 增加人工重点事件入口
- 实现分析融合引擎第一版

完成标准：

- 对重大事件能输出结构化影响，而不是只做摘要

## Phase 4: 基本面支持层与评估层

目标：

- 提升建议质量，建立优化闭环

任务：

- 增加轻基本面支持
- 建立分层评估报表
- 区分不同类型建议的真实效果
- 建立“小资金高赔率机会”专项评估视图

完成标准：

- 可以明确知道哪些建议家族有效，哪些无效

## 9. 评估体系设计

这是后续最不能省的一部分。

建议至少按以下维度评估：

### 9.1 市场层评估

- 风险模式判断正确率
- 风格判断正确率
- 情绪周期判断后的环境适配收益差异

### 9.2 主线层评估

- 主线持续性
- 龙头识别准确度
- 跟风回避是否有效

### 9.3 个股层评估

- `buy_pilot` 命中率
- `observe` 转正率
- `avoid` 回避效果
- 高弹性机会的平均收益空间
- 大票低弹性机会是否占用了过多注意力

### 9.4 模块归因评估

按建议来源拆解：

- 事件驱动
- 宏观驱动
- 主题驱动
- 技术驱动
- 混合共振

### 9.5 执行层评估

- 是否因为账户限制而规避了明显不适合的交易
- 是否因为过度保守错失高质量机会

### 9.6 小资金目标专项评估

必须单独回答：

- 系统是否真的把机会优先分配给高赔率方向
- 是否过度偏向“安全但不赚钱”的建议
- 是否在可接受回撤下提升了收益弹性
- 龙头/前排建议是否优于中庸票建议

## 10. 最终形态判断

如果把系统做到最好，我认为它最合理的最终形态不是“自动赚钱机器”，而是：

一个面向 A 股个人交易者的、分层清晰的、事件驱动和情绪驱动优先、技术确认辅助、账户适配严格、长期可评估迭代的交易分析系统。

它的核心竞争力不应该是：

- 神奇预测

而应该是：

- 比纯主观更稳定
- 比纯技术更懂市场语境
- 比普通资讯摘要更接近交易决策
- 比杂乱的人工盯盘更可复盘和可优化
- 对小资金账户更聚焦高赔率机会，而不是输出一套偏机构化的低弹性建议

## 11. 当前设计结论

一句话总结：

下一阶段不是继续堆技术指标，而是要把系统从“结构化盘前分析器”升级成“市场状态 + 宏观事件 + 主线映射 + 技术确认 + 评估闭环”的综合决策系统。

后续所有开发，都应优先服务这个目标。
