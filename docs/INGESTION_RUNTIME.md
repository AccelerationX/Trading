# Ingestion Runtime

## 1. 目标

本文件说明 `TradingSystem` 第一版每日信息接入运行方式。

当前阶段不直接实现所有外部数据抓取器，而是先实现统一接入框架，解决以下问题：

- 今天系统应该检查哪些来源
- 每个来源的数据是否已经到位
- 原始输入文件是否被快照保存
- 每日接入清单是否可追溯

## 2. 当前运行模式

第一版采用两种连接方式：

- `file_drop`
  - 外部数据先落到本地目录
  - 系统扫描目录、记录文件、复制到每日快照
- `manual`
  - 人工填写或人工放入的输入
  - 系统同样做发现、归档和快照

这意味着即使外部 API 还没接，整个接入流水线已经可以跑起来。

## 3. 目录约定

### 3.1 输入收件箱

```text
data/inbox/<source_id>/
```

### 3.2 每日快照

```text
data/snapshots/YYYY-MM-DD/<source_id>/
data/snapshots/YYYY-MM-DD/manifest.json
```

## 4. 运行流程

每日接入流程：

1. 读取来源注册表。
2. 读取来源连接配置。
3. 读取每日接入计划。
4. 检查每个来源的输入目录和文件。
5. 复制当日发现的文件到快照目录。
6. 生成 `manifest.json`。
7. 输出摘要，提示哪些核心来源缺失。

## 5. manifest 的作用

`manifest.json` 是每日接入证据链的核心文件，用来回答：

- 今天系统看了哪些来源
- 哪些来源到位了
- 哪些来源为空或缺失
- 快照目录里保存了哪些原始文件

## 6. 当前脚手架能力

当前脚手架支持：

- A 类来源的本地接入配置模板
- 来源目录扫描
- 原始文件快照复制
- manifest 落盘
- 命令行运行入口
