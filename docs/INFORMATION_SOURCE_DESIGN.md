# Information Source Design

## 1. 目标

本文件定义 `TradingSystem` 第一版信息源体系，解决两个问题：

1. 系统到底应该收哪些信息。
2. 每类信息进入系统后，应该服务什么判断。

这里先定“信息类别和规则”，不急着先定具体站点、接口和抓取脚本。

## 2. 信息源设计原则

### 2.1 先看交易价值，不先看数量

不是信息越多越好。  
一个信息源只有在能明显帮助以下至少一个问题时，才值得进入系统：

- 今天是否适合交易
- 哪个主线/行业正在增强或退潮
- 哪只股票进入重点候选
- 何时进场、何时减仓、何时退出

### 2.2 先收可解释的信息，再收“神秘 alpha”

优先接入能够直接解释的信号，例如：

- 公告
- 政策
- 行业催化
- 涨停结构
- 龙虎榜
- 解禁
- 业绩变化

暂时不优先接入那些“看起来有用，但解释链很弱”的数据源。

### 2.3 先保证时点真实，再讨论效果

任何进入系统的信息都要尽量保留：

- 发布时间
- 获取时间
- 来源
- 是否盘前可得 / 盘中可得 / 盘后才可得

否则就会以另一种形式重演未来函数问题。

### 2.4 信息源必须服务某一层，而不是混着用

每类信息都要明确归属到以下至少一层：

- `市场环境层`
- `事件理解层`
- `候选筛选层`
- `仓位与执行层`
- `经验复盘层`

### 2.5 先建立源分级，再接入具体来源

对每类信息先定义：

- 作用
- 优先级
- 可信度
- 时效要求
- 是否需要 LLM 参与解释

## 3. 信息源评价维度

后续每增加一个具体信息源，都按这几项打标签：

- `trade_value`
  - 对交易决策帮助大 / 中 / 小
- `timeliness`
  - 盘前 / 盘中 / 盘后 / 低频
- `credibility`
  - 官方 / 交易所 / 公司 / 主流财经媒体 / 二手转载 / 社交噪音
- `structure_level`
  - 结构化 / 半结构化 / 文本型
- `llm_dependency`
  - 不需要 LLM / 适合 LLM 摘要 / 必须依赖 LLM 解释
- `historical_replay`
  - 后续是否适合做历史快照与复盘验证

## 4. 第一版信息源分层

## 4.1 A 类：必须收集

这些是系统第一阶段必须有的信息。

### A1. 市场行情与结构

内容包括：

- 个股日线行情
- 指数行情
- 板块/行业行情
- 成交额、换手、量比
- 涨停、跌停、炸板
- 连板高度
- 涨跌家数
- 市场总成交额

主要用途：

- 判断今天适不适合交易
- 判断是进攻、轮动、防守还是兑现环境
- 识别主线强度和情绪变化
- 给个股建议提供市场背景

对应模块：

- `ingest`
- `context`
- `signal`

是否需要 LLM：

- 不一定必须
- LLM 主要用于解释，不用于计算原始结构指标

### A2. 公司公告与正式事件

内容包括：

- 业绩预告、业绩快报
- 增减持
- 回购
- 解禁
- 重组并购
- 停复牌
- 重大合同
- 中标公告
- 监管关注、立案、处罚

主要用途：

- 判断个股是否存在明确催化或风险
- 识别短中期事件驱动
- 识别利空避险对象
- 给个股建议增加“为什么是它”的证据链

对应模块：

- `ingest`
- `integrations`
- `context`
- `decision`

是否需要 LLM：

- 需要
- LLM 负责把公告内容转成事件类型、利多利空方向、持续性和行业映射

### A3. 政策与行业催化

内容包括：

- 部委政策
- 行业监管动向
- 产业扶持
- 国资、地方政策
- 行业会议、产业大会、技术突破消息

主要用途：

- 识别大级别主线
- 判断哪些行业可能进入一段行情
- 区分短脉冲消息和中期催化

对应模块：

- `integrations`
- `context`
- `decision`

是否需要 LLM：

- 强依赖
- 这类信息核心难点不是采集，而是解释“利好谁、利好多久、逻辑链条是什么”

### A4. 资金行为与交易痕迹

内容包括：

- 龙虎榜
- 大宗交易
- 北向资金
- 融资融券变化
- 异常成交与换手结构

主要用途：

- 判断资金是否真实参与
- 识别强势股是否有承接
- 判断题材是否只是消息刺激，还是有资金配合
- 辅助进场、减仓和回避高风险个股

对应模块：

- `ingest`
- `context`
- `decision`

是否需要 LLM：

- 中度依赖
- 原始统计靠程序，解释资金意图和风险由 LLM 辅助

### A5. 基础公司与板块映射表

内容包括：

- 股票基础信息
- 所属行业
- 所属概念
- 指数成分
- 流通市值和基本分类

主要用途：

- 把事件映射到行业和个股
- 支持“这个利好到底影响哪一批股票”
- 支持观察池和主线池构建

对应模块：

- `ingest`
- `signal`
- `context`

是否需要 LLM：

- 不强依赖
- 更偏基础参考表

### A6. 交易者个人账户与约束信息

内容包括：

- 总资金
- 单票仓位上限
- 持仓只数上限
- 可接受回撤
- 可接受换手
- 盯盘时间和执行偏好

主要用途：

- 让建议真正适配你，而不是适配一个抽象账户
- 限制理想化高换手和过度分散
- 决定买入数量、减仓方式和退出节奏

对应模块：

- `config`
- `decision`
- `memory`

是否需要 LLM：

- 不强依赖
- 但 LLM 需要读取这些约束来改写建议

## 4.2 B 类：有价值，但第二阶段再加

这些信息很有价值，但不该在第一轮就把系统做复杂。

### B1. 新闻快讯与财经媒体报道

价值：

- 帮助更早感知题材和事件扩散
- 可辅助公告之外的市场解释

风险：

- 重复转载多
- 标题党严重
- 噪音比例高

建议：

- 第二阶段再正式接入
- 接入时必须做去重、来源分级和摘要压缩

### B2. 分析师研报与机构观点

价值：

- 可以帮助理解行业逻辑和市场预期差

风险：

- 时效不稳定
- 可获得性和历史回放困难
- 观点质量差异大

建议：

- 第二阶段末或第三阶段再考虑

### B3. 更细的基本面数据

内容包括：

- 更完整财报明细
- 估值历史分位
- 机构持仓

价值：

- 对中期择股有帮助

风险：

- 对短线和事件决策未必是刚需
- 先接会拖慢主流程设计

### B4. 盘中细粒度分时与盘口结构

价值：

- 对进场时机和盘中确认有帮助

风险：

- 容易把系统拖进高频伪精细化
- 对人工执行和复盘要求很高

建议：

- 等日级系统跑通后再考虑

## 4.3 C 类：暂时不建议碰

### C1. 社交平台热帖与论坛情绪

原因：

- 噪音极大
- 易被操纵
- 时点难核

除非后续建立严格可信度体系，否则先不进主系统。

### C2. 过度复杂的另类数据

例如：

- 模糊来源资金热度
- 无法验证采集时点的“神秘内部榜单”
- 无法长期稳定获取的数据

原因：

- 极易引入伪优势
- 难复盘
- 难解释

### C3. 依赖强主观二次加工且无法归档的口口相传信息

原因：

- 不能审计
- 不能回放
- 不能验证真伪

## 5. 每类信息进入系统后的中间产物

系统不应该直接把“原始消息”喂给最终决策，而应该先结构化。

### 5.1 市场行情与结构 -> `market_regime_snapshot`

示例字段：

- market_bias
- risk_on_off
- breadth_strength
- limit_up_temperature
- turnover_regime
- main_style

### 5.2 公告/事件 -> `event_cards`

示例字段：

- event_type
- source
- publish_time
- bullish_bearish
- impact_scope
- duration_guess
- related_industries
- related_stocks
- confidence

### 5.3 政策/行业催化 -> `theme_cards`

示例字段：

- theme_name
- trigger_source
- policy_level
- likely_beneficiary_chain
- immediate_beneficiaries
- medium_term_beneficiaries
- market_confirmation_needed

### 5.4 资金行为 -> `capital_behavior_cards`

示例字段：

- capital_signal_type
- participation_strength
- consistency
- speculative_or_institutional_guess
- warning_flags

### 5.5 候选股票池 -> `candidate_pool`

示例字段：

- stock_code
- candidate_reason
- technical_state
- event_support
- market_fit
- account_fit

### 5.6 最终交易建议 -> `trade_plan_cards`

示例字段：

- action
- stock_code
- rationale
- entry_condition
- position_size_rule
- invalidation_rule
- exit_hint
- risk_notes

## 6. 信息可信度分级

为了避免消息噪音污染系统，定义四级来源：

- `S`
  - 交易所、上市公司原文、官方政策原文、官方统计口径
- `A`
  - 高质量主流财经媒体、可核验正规资讯源
- `B`
  - 二次整理平台、摘要类媒体
- `C`
  - 社交传闻、论坛、未经核验自媒体

规则：

- `S/A` 可进主分析流
- `B` 只能做参考，不能单独触发交易建议
- `C` 默认不进主分析流

## 7. 时间属性规则

每条信息都要尽量打上以下标签：

- `known_time`
  - 系统何时获取到
- `publish_time`
  - 信息对外发布时间
- `market_phase`
  - 盘前 / 盘中 / 盘后 / 周末
- `decision_scope`
  - 可用于今日盘前 / 仅能用于次日 / 仅用于复盘

这是为了避免日后在建议生成中出现时点污染。

## 8. 第一阶段建议的实际收集顺序

建议顺序如下：

1. 市场行情与结构
2. 公司公告与正式事件
3. 政策与行业催化
4. 资金行为与交易痕迹
5. 基础映射表
6. 账户约束与人工输入

原因：

- 这 6 类已经足以支持“市场环境 + 主线识别 + 个股建议 + 仓位建议”的第一版闭环。

## 9. 当前不做的事

第一阶段不做：

- 海量新闻全文库
- 社交情绪抓取
- 高频盘口策略
- 黑盒情绪评分
- 无法回放来源时间的外部热榜

## 10. 下一轮讨论重点

在本文件基础上，下一轮应讨论：

1. A 类信息里，每一类到底优先找哪些具体来源。
2. 哪些信息可以先手工输入，哪些必须自动采集。
3. 每日建议最终展示时，要把哪几类中间产物呈现给你。
