# TradingSystem 总设计

## 1. 目标定义

`TradingSystem` 的目标不是做一个“全自动 AI 炒股程序”，而是做一个适用于 A 股个人交易者的决策辅助系统。

系统每天要回答 5 个核心问题：

1. 今天这个市场环境适不适合交易。
2. 今天最值得关注的主线、题材、行业或事件是什么。
3. 在当前环境下，哪些股票最值得进入候选池。
4. 如果要交易，应该怎么进、进多少、何时减、何时退。
5. 如果判断错误，什么条件算失效，应该如何止损或放弃。

## 2. 为什么要单独建新系统

旧项目的问题已经比较清楚：

- `Trading` 的主要问题是历史研究中混入未来数据、成交假设失真、回测收益虚高。
- `TradingMain` 的主要问题是虽然更严谨，但仍偏理想化，且对真实资金规模、执行摩擦和消息语境适配不够。
- 两者共同问题是过度依赖历史价格和因子组合，对实时信息、政策环境、题材扩散和交易者个人经验利用不足。

因此新系统必须从一开始就坚持：

- 历史研究只是底座，不是最终裁判。
- LLM 负责解释与整合，不直接代替人做黑盒决策。
- 所有建议都要落到真实账户约束上。

## 3. 系统边界

### 3.1 系统要做的事

- 聚合和整理结构化、半结构化、非结构化信息。
- 生成市场环境判断、主线判断、个股候选和交易建议。
- 把量化候选、事件信息、风险约束、个人经验综合起来。
- 输出每日操作建议、观察池、风险提示和复盘材料。

### 3.2 系统不做的事

- 不自动下单。
- 不允许无法解释来源的纯黑盒建议。
- 不把未经时点校验的未来信息混入实时决策流程。
- 不把“回测好看”当成系统有效的唯一标准。

## 4. 核心设计原则

### 4.1 人机协同

系统生成建议，人工做最终执行。  
LLM 是研究助手、事件分析助手、决策辅助助手，不是独立交易员。

### 4.2 证据链

每条建议必须尽量带上：

- 触发逻辑
- 相关信息源
- 所属主线或事件
- 风险点
- 失效条件
- 仓位理由

### 4.3 真实约束优先

建议必须考虑：

- 总资金规模
- 单票仓位上限
- 持仓数量上限
- 可接受换手
- 可接受回撤
- 交易者是否有时间盯盘

### 4.4 分层架构

不同职责拆分为独立层，避免旧项目那种研究脚本、实时输出、用户操作台混在一起。

## 5. 系统分层

### 5.1 数据层

职责：

- 存放市场数据、公告、新闻、政策、资金、参考表和人工补录信息。
- 管理原始数据与处理中间数据。
- 保留时点快照，方便后续验证建议是否使用了当时可得信息。

目录承接：

- `data/raw/`
- `data/processed/`
- `data/snapshots/`

### 5.2 研究层

职责：

- 存放假设、实验、复盘、方法论修订。
- 承接旧 `Trading` / `TradingMain` 的历史经验，但不直接复制旧结构。
- 把“为什么这么设计”记录下来，而不只是保留代码。

目录承接：

- `research/hypotheses/`
- `research/experiments/`
- `research/reviews/`

### 5.3 候选生成层

职责：

- 对接历史量化研究成果。
- 生成候选股票池、基础风险过滤、基本技术状态描述。
- 不直接输出最终交易建议。

代码承接：

- `src/trading_system/signal/`

### 5.4 信息理解层

职责：

- 接收新闻、公告、政策、事件、板块异动等信息。
- 用 LLM 做事件抽取、行业映射、情绪判断、持续性判断、风险标签。
- 把噪音消息转成结构化观点。

代码承接：

- `src/trading_system/context/`
- `src/trading_system/integrations/`
- `prompts/`

### 5.5 决策融合层

职责：

- 结合候选池、市场环境、事件判断、账户约束和经验记忆。
- 生成买卖建议、仓位建议、观察名单和放弃条件。

代码承接：

- `src/trading_system/decision/`

### 5.6 经验记忆层

职责：

- 记录人工交易、理由、结果、复盘结论。
- 形成“过去在类似局面下怎么做、结果如何”的可检索记忆。
- 为后续 LLM 决策提供个人风格和经验约束。

代码承接：

- `src/trading_system/memory/`
- `workspace/journal/`

### 5.7 输出层

职责：

- 生成每日操作建议、观察池、交易日报、风险摘要、复盘材料。

目录承接：

- `outputs/daily_reports/`
- `outputs/trade_plans/`
- `outputs/analysis/`

## 6. 目标用户工作流

预期日常流程：

1. 数据更新与快照固化。
2. 系统判断市场环境。
3. 系统识别主线、事件和重点行业。
4. 系统从候选池中筛出重点股票。
5. 系统生成交易建议。
6. 交易者人工审阅并执行。
7. 交易后补录真实成交与主观判断。
8. 系统在收盘后生成复盘与经验沉淀。

## 8. 代码模块职责

### `config`

- 全局路径
- 环境配置
- 参数与账户约束
- 信息源配置

### `ingest`

- 原始数据读取
- 数据标准化
- 快照写入

### `integrations`

- 外部数据源接入
- 消息源适配器
- 公告、新闻、行情等连接层

### `context`

- 市场环境判断
- 事件解析
- 主题/行业关联分析
- LLM 结构化输出

### `signal`

- 历史策略线接入
- 候选股票池生成
- 因子/技术状态摘要

### `decision`

- 候选合并
- 仓位分配
- 买卖规则融合
- 建议生成

### `memory`

- 交易记录结构
- 经验案例检索
- 复盘总结入库

### `reporting`

- 日报
- 操作建议卡片
- 复盘文档

## 9. 需要重点防止的失败模式

- 把 LLM 当成会自动预测涨跌的黑盒。
- 把消息摘要直接当成交易结论。
- 把旧量化高收益结果原样迁入新系统。
- 忽略资金规模和成交约束。
- 在盘后复盘和盘前决策之间混入时点错位。
- 没有记录人工最终为什么执行或不执行建议。

## 10. 第一阶段产出要求

在正式做信息源设计前，本项目需要先把以下约束定死：

- 系统不自动下单。
- 每日建议必须可解释。
- 每条建议必须含有风险提示和失效条件。
- 所有外部信息必须尽量记录来源与时间。
- 人工交易结果必须能够回流到经验记忆层。

## 11. 当前未定事项

这些问题后续专门讨论：

- 第一批信息源到底收哪些。
- 每日建议的最终展示格式。
- 账户约束的默认规则。
- 旧 `Trading` / `TradingMain` 哪些研究成果值得迁入。
- 经验记忆的数据结构如何定义。
